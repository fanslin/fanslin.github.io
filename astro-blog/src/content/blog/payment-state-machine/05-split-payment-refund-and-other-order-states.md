## 一个支付场景问题

支付成功后，用户申请退款。你会不会把支付单状态从 `PAID` 改成 `REFUNDED`？

如果只是全额退款，这么做似乎还能解释。但如果是部分退款呢？如果一笔支付单发生多次退款呢？如果后续还要请款、拒付、结算呢？

把所有业务状态都塞进支付主单，很快会把状态机变成一张难以维护的大网。

## 支付主单只管支付生命周期

支付主单最核心的生命周期其实很短：

```text
INIT -> PAYING -> PAID/FAILED
```

`PAID` 表示支付成功。对支付主单来说，这已经是终态。

退款不是支付主单生命周期的后续状态，而是另一个业务对象的生命周期。

## 为什么 REFUNDED 不适合放进支付主单

把 `REFUNDED` 放进支付主单，会带来几个问题。

第一，部分退款不好表达。一笔支付 100 元，退款 30 元，支付单到底是 `PAID` 还是 `REFUNDED`？

第二，多次退款不好表达。一笔支付可能发生多笔退款单，每笔退款都有自己的处理中、成功、失败状态。

第三，职责混乱。支付单负责支付成功与失败，退款单负责退款成功与失败。两个生命周期混在一起，后续代码会越来越绕。

第四，对账和排障困难。支付成功和退款成功是两个不同事实，应该分别记录。

## 更清晰的拆分方式

可以把交易系统拆成多个单据，每个单据有自己的状态机：

| 单据 | 负责的生命周期 | 示例状态 |
| --- | --- | --- |
| 支付单 | 支付结果 | `INIT`、`PAYING`、`PAID`、`FAILED` |
| 退款单 | 退款结果 | `REFUNDING`、`REFUNDED`、`REFUND_FAILED` |
| 请款单 | 预授权后的请款结果 | `CAPTURING`、`CAPTURED`、`CAPTURE_FAILED` |
| 拒付单 | 持卡人拒付处理 | `DISPUTING`、`DISPUTE_WON`、`DISPUTE_LOST` |

这样状态机数量变多了，但每个状态机都更小、更清楚。

## 一个概念设计

支付主单状态机：

```text
null   + PAY_CREATE  -> INIT
INIT   + PAY_PROCESS -> PAYING
PAYING + PAY_SUCCESS -> PAID
PAYING + PAY_FAIL    -> FAILED
```

退款单状态机：

```text
null      + REFUND_CREATE  -> REFUNDING
REFUNDING + REFUND_SUCCESS -> REFUNDED
REFUNDING + REFUND_FAIL    -> REFUND_FAILED
```

支付主单到 `PAID` 后，可以创建退款单。但退款单状态变化不应该回写成支付主单状态。

## 领域关系怎么表达

支付单和退款单之间可以通过业务关联表达：

```text
payment_id = PAY-1001
refund_id  = REF-2001
```

退款单引用支付单，但不吞掉支付单的状态。

支付单回答：“这笔支付是否成功？”

退款单回答：“这笔退款是否成功？”

这两个问题不应该混成一个状态字段。

## 本篇小结

状态机不是越大越完整。交易系统里更推荐“一个单据，一个清晰生命周期”。

支付单只管支付，退款单只管退款，请款单只管请款，拒付单只管拒付。这样状态机虽然变多，但每一个都更容易理解、测试和排障。
