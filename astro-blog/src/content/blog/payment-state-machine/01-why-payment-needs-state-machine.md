## 一个支付场景问题

假设用户发起支付后，渠道先发来异步通知：支付成功。你的系统把支付单更新成 `PAID`。

几百毫秒后，同步接口返回来了，但它返回的是“支付中”。如果代码只是简单地按返回结果直接改状态，那么这笔单可能会从 `PAID` 又被改回 `PAYING`。

这不是理论问题。支付系统天然会遇到同步返回、异步通知、重复通知、乱序通知、网络超时、定时查单、人工补单。只要状态更新没有被统一约束，系统就可能被旧消息或重复消息带偏。

## 状态不是普通字段

支付单的状态不是一个随便赋值的字符串。它代表一笔交易当前处于哪个事实阶段：

- `INIT`：支付单已经创建。
- `PAYING`：正在等待渠道结果。
- `PAID`：支付成功，通常是支付主单终态。
- `FAILED`：支付失败，通常也是终态。

一旦进入终态，就不应该被旧消息改回中间态。这个边界如果只靠开发人员自觉，很容易在后续需求里被破坏。

状态机的价值，就是把这些边界显式写出来。

## 直接改状态的问题

很多业务代码一开始会这样写：

```java
paymentModel.setStatus(PaymentStatus.PAID);
```

这行代码看起来简单，但它绕过了几个关键问题：

- 当前状态是否允许变成 `PAID`？
- 这个变化是哪个业务事件触发的？
- 如果当前已经是 `FAILED`，还能不能改成 `PAID`？
- 如果当前已经是 `PAID`，又收到 `PAY_PROCESS`，应该怎么办？

直接 `setStatus` 的最大问题是：状态变化散落在业务代码各处。后来排查问题时，只能全项目搜索谁改过状态。

## 状态机的核心表达

状态机把状态变化收束成一个规则：

```text
当前状态 + 事件 -> 目标状态
```

例如：

```text
INIT   + PAY_PROCESS -> PAYING
PAYING + PAY_SUCCESS -> PAID
PAYING + PAY_FAIL    -> FAILED
```

业务代码不再表达“我要把状态改成成功”，而是表达“支付成功事件发生了”：

```java
paymentModel.transferStatusByEvent(PaymentEvent.PAY_SUCCESS);
```

状态机负责判断这个事件在当前状态下是否合法。

## 状态机带来的直接收益

第一，规则集中。看支付单怎么流转，不需要翻业务代码，只看状态机定义。

第二，非法推进会被拒绝。比如 `INIT + PAY_SUCCESS` 没有定义，就不能直接成功。

第三，终态不会被回退。比如 `PAID` 后没有任何后续转移，旧的 `PAY_PROCESS` 事件无法把它改回 `PAYING`。

第四，排障更清楚。每次状态变化都可以记录原状态、事件、目标状态、结果和失败原因。

## 本篇小结

支付系统需要状态机，不是因为状态机这个模式很优雅，而是因为支付链路充满异步、重复、乱序和并发。

状态机的第一价值是保护交易事实：什么状态可以接收什么事件，什么状态已经结束，不能再被回退。
