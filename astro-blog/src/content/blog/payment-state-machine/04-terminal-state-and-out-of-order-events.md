## 一个支付场景问题

支付成功后，系统已经把支付单更新成 `PAID`。

但稍后又来了一个旧消息：`PAY_PROCESS`。它可能来自同步返回、消息重试、缓存延迟，或者某个慢请求。

这时应该怎么办？

答案很明确：不能把 `PAID` 改回 `PAYING`。

## 什么是终态

终态表示这个业务对象在当前生命周期里已经结束。对支付主单来说：

- `PAID`：支付成功。
- `FAILED`：支付失败。

进入终态后，支付主单状态不应该再被中间状态覆盖。

如果后续发生退款，不应该把支付主单改成 `REFUNDED`，而应该创建退款单，由退款单维护自己的生命周期。

## 乱序消息为什么常见

支付链路不是单线程顺序执行。常见情况包括：

- 渠道异步通知先到，同步接口后返回。
- 消息队列重复投递。
- 定时查单和异步通知同时处理同一笔单。
- 网络抖动导致旧请求晚到。
- 人工补单和自动补偿同时发生。

所以不能假设“后到的消息一定更新”。后到的消息可能已经过期。

## 状态机如何防回退

当前 demo 里，合法规则只有：

```text
null   + PAY_CREATE  -> INIT
INIT   + PAY_PROCESS -> PAYING
PAYING + PAY_SUCCESS -> PAID
PAYING + PAY_FAIL    -> FAILED
```

注意，没有这条规则：

```text
PAID + PAY_PROCESS -> PAYING
```

所以当支付单已经是 `PAID`，再收到 `PAY_PROCESS`，状态机查询不到目标状态，会抛出异常：

```text
状态转换失败: currentStatus=PAID, event=PAY_PROCESS
```

这就是终态防回退。

## 拒绝非法事件不是系统错误

很多人看到异常会本能地认为“系统出错了”。但在状态机里，拒绝非法事件是正常保护行为。

比如：

```text
PAID + PAY_PROCESS
```

这代表一个旧消息试图回退状态。拒绝它，才是正确行为。

生产系统里可以把这种结果细分为：

- `REJECTED`：非法推进。
- `IDEMPOTENT`：重复事件，状态保持不变。
- `SUCCESS`：合法推进。
- `RETRYABLE`：暂时失败，可重试。

当前 demo 用异常表达拒绝，是为了让入门实现更直接。

## 支付成功后同步返回“支付中”的案例

一个典型时序是：

```text
T1 用户发起支付，支付单 INIT -> PAYING
T2 渠道异步通知成功，支付单 PAYING -> PAID
T3 同步接口返回支付中，旧事件 PAY_PROCESS 到达
T4 状态机发现当前是 PAID，拒绝 PAY_PROCESS
```

如果没有状态机，T3 很可能把状态改回 `PAYING`。

有了状态机，T3 只是一个被拒绝的旧事件。

## 本篇小结

支付系统不能按“消息到达顺序”决定最终状态。最终状态必须由“当前状态 + 事件是否合法”决定。

状态机最大的实际价值之一，就是保护终态不被旧消息、重复消息、慢请求回退。
