## 一个支付场景问题

同一笔支付单，同时来了两个线程：

- 线程 A 处理渠道异步通知：`PAY_SUCCESS`
- 线程 B 处理定时查单结果：`PAY_SUCCESS`

它们都读到当前状态是 `PAYING`，状态机也都判断可以推进到 `PAID`。

问题来了：状态机判断合法，是否就代表数据库一定安全？

答案是否定的。

## 状态机解决规则合法性

状态机负责判断：

```text
PAYING + PAY_SUCCESS -> PAID
```

这条转移是否合法。

但状态机通常运行在内存里。两个线程可能同时读到同一个旧状态，也可能分别在不同机器上处理同一笔消息。

所以状态机只能回答“按规则能不能转”，不能单独保证“只有一个线程转成功”。

## 数据库解决并发写入唯一性

支付核心状态最终要落数据库。并发场景下，常见做法是乐观锁：

```sql
update payment_order
set status = ?, version = version + 1
where payment_id = ?
  and status = ?
  and version = ?
```

这里的关键是 `status` 和 `version` 条件。

只有数据库当前状态和版本号仍然匹配时，更新才会成功。否则说明已经有别的线程先更新了。

## 状态机和数据库怎么配合

一个常见流程是：

```text
1. 查询支付单，拿到 currentStatus 和 version
2. 状态机判断 currentStatus + event 是否有合法 targetStatus
3. 执行带 status/version 条件的 update
4. update 成功：记录流转日志，提交后续消息
5. update 失败：重新查询当前状态，判断是幂等成功、非法推进，还是需要重试
```

状态机和数据库不是替代关系，而是配合关系：

- 状态机保证规则正确。
- 数据库保证并发更新只有一个成功。

## 幂等：重复通知不一定是错误

支付系统里，重复通知是常态。比如支付单已经是 `PAID`，又收到一次 `PAY_SUCCESS`。

这类情况可以设计为幂等成功：

```text
PAID + PAY_SUCCESS -> PAID
result = IDEMPOTENT
```

注意，幂等不等于所有重复事件都放行。比如：

```text
PAID + PAY_FAIL
```

这通常不应该幂等成功，而应该拒绝或进入差错处理。

## 幂等策略要显式

不要把幂等写成随意判断：

```java
if (currentStatus == targetStatus) {
    return;
}
```

更好的做法是让状态机或 `Transition` 显式声明哪些事件可以幂等：

```text
PAID + PAY_SUCCESS -> PAID，允许幂等
FAILED + PAY_FAIL -> FAILED，允许幂等
PAID + PAY_FAIL -> 拒绝
```

这样业务含义更清楚，也更容易审计。

## 失败后的处理

带版本号更新失败后，不要马上判定系统异常。应该重新读一次数据库状态。

如果数据库已经是 `PAID`，而当前事件也是 `PAY_SUCCESS`，可以视为幂等成功。

如果数据库已经是 `PAID`，但当前事件是 `PAY_FAIL`，说明渠道结果冲突，需要进入差错处理或人工核查。

如果数据库仍是 `PAYING`，可能是临时冲突或网络问题，可以重试。

## 补偿仍然不可少

并发控制解决的是内部一致性，不能解决外部渠道不确定性。

所以支付系统仍然要有：

- 查单补偿。
- 消息重投。
- 对账文件。
- 差错处理。

状态机让补偿结果也必须经过合法状态推进，避免补偿代码绕过核心规则。

## 本篇小结

状态机只解决“规则合法性”，不能单独解决“并发写入唯一性”。

生产支付系统需要把状态机、数据库乐观锁或悲观锁、幂等策略、补偿机制组合起来。只有这样，才能在重复通知、乱序消息和多线程处理中保持最终状态正确。
